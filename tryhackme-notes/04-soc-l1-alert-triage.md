# SOC L1 Alert Triage

## Overview

This note summarizes the concept of **SOC alerts** and the **alert triage workflow** for a SOC L1 Analyst.

An alert is one of the core concepts in a SOC environment.

Knowing how to handle alerts properly can decide whether a security breach is detected and prevented, or missed and allowed to cause serious damage.

Alert triage is an essential responsibility for SOC L1 Analysts.

---

## What Is an Alert?

An **alert** is a notification generated by a security solution when a specific event or sequence of events matches suspicious, anomalous, or rule-based behavior.

Examples of alert names:

- Email Marked as Phishing
- Unusual Gmail Login Location
- Windows RDP Bruteforce
- Potential Data Exfiltration
- Double-Extension File Creation
- Unapproved Mimikatz Usage

Alerts help SOC Analysts focus on suspicious activity instead of manually reviewing millions of raw logs.

---

## From Events to Alerts

Before an alert exists, an event must happen.

Example events:

- User login
- Process launch
- File download
- Network connection
- Failed authentication
- Email delivery
- Command execution

The event is then logged by a system.

Examples of log sources:

- Operating system
- Firewall
- Cloud provider
- Endpoint security tool
- Application
- Authentication system
- Mail platform

After that, logs are shipped to a security solution such as a SIEM, EDR, NDR, or SOAR.

If the security solution detects suspicious behavior, it generates an alert.

Basic flow:

**Event → Log → Security Tool → Alert → Triage → Classification → Escalation or Closure**

---

## Raw Logs vs Alerts

A **raw log** records an event.

An **alert** highlights an event or sequence of events that may require security investigation.

Raw logs can be extremely large in volume.

A SOC team may receive millions of logs per day from many systems.

Alerts reduce this workload by highlighting activity that may be suspicious, anomalous, or malicious.

**Analyst sentence:**

A raw log records what happened, while an alert highlights activity that may require investigation.

---

## Alert Management Platforms

Alerts can appear in different security or ticketing platforms.

### SIEM

**SIEM** stands for **Security Information and Event Management**.

Examples:

- Splunk Enterprise Security
- Elastic

SIEM platforms collect and correlate logs from multiple sources and generate alerts based on detection rules.

SIEM is often the central alert management platform for SOC teams.

### EDR / NDR

**EDR** stands for **Endpoint Detection and Response**.

**NDR** stands for **Network Detection and Response**.

Examples:

- Microsoft Defender
- CrowdStrike

EDR and NDR can provide their own alert dashboards.

However, in many SOC environments, EDR and NDR alerts are also sent to SIEM or SOAR for central handling.

### SOAR

**SOAR** stands for **Security Orchestration, Automation, and Response**.

Examples:

- Splunk SOAR
- Cortex SOAR

SOAR platforms can aggregate alerts from multiple solutions and automate parts of the response process.

### ITSM

**ITSM** stands for **IT Service Management**.

Examples:

- Jira
- TheHive

Some SOC teams use ITSM or ticketing platforms to manage alert investigation, case ownership, and response workflows.

---

## L1 Role in Alert Triage

SOC L1 Analysts are usually the first line of defense in the alert triage process.

They work with alerts more than anyone else in the SOC team.

A SOC L1 Analyst may receive zero to a hundred alerts per day depending on the organization, tooling, and threat activity.

The main responsibility of a SOC L1 Analyst is to review alerts and decide whether the activity is benign, suspicious, or malicious.

### SOC L1 Analyst Responsibilities

A SOC L1 Analyst should:

- Review alerts
- Understand alert details
- Check key fields
- Distinguish benign activity from suspicious activity
- Identify possible real threats
- Document findings
- Escalate real threats to L2 analysts
- Close false positives when appropriate

### SOC L2 Analyst Role

SOC L2 Analysts receive alerts escalated by L1 Analysts.

They perform deeper analysis, correlate more data sources, and support remediation.

### SOC Engineer Role

SOC Engineers make sure alerts contain enough useful information for efficient triage.

They help improve alert quality, parsing, enrichment, and detection logic.

### SOC Manager Role

SOC Managers track the speed and quality of alert triage.

They make sure real attacks are not missed and that the SOC process works properly.

## Alert Properties

Alerts generally contain several important properties.

A SOC Analyst must understand these fields before taking action.

### Alert Time

**Alert Time** shows when the alert was created.

The alert time may be a few minutes later than the actual event time.

Example:

- Event Time: March 21, 15:32
- Alert Time: March 21, 15:35

### Alert Name

**Alert Name** summarizes what happened.

It is usually based on the detection rule name.

Examples:

- Unusual Login Location
- Email Marked as Phishing
- Windows RDP Bruteforce
- Potential Data Exfiltration
- Double-Extension File Creation

### Alert Severity

**Alert Severity** defines the urgency or importance of the alert.

Severity is often set by detection engineers but may sometimes be adjusted by analysts.

Common severity levels:

- Low / Informational
- Medium / Moderate
- High / Severe
- Critical / Urgent

### Alert Status

**Alert Status** shows where the alert is in the workflow.

Common statuses:

- New
- Unassigned
- In Progress
- Pending
- Awaiting Action
- Closed
- Resolved

Status helps prevent multiple analysts from working on the same alert at the same time.

### Alert Verdict

**Alert Verdict** is also called **alert classification**.

It explains whether the alert was a real threat or noise.

Common verdicts:

- True Positive
- False Positive

**True Positive** means the alert represents a real threat or suspicious malicious activity.

**False Positive** means the alert does not represent a real threat.

### Alert Assignee

**Alert Assignee** shows the analyst responsible for reviewing the alert.

The assignee can also be called the alert owner.

Assigning an alert creates ownership and prevents duplicate work.

### Alert Description

**Alert Description** explains what the alert is about.

It may include:

- The logic of the detection rule
- Why the activity may indicate an attack
- How the alert should be triaged
- Related investigation guidance

### Alert Fields

**Alert Fields** provide the key evidence that triggered the alert.

Examples:

- Affected hostname
- Username
- Source IP
- Destination IP
- Process name
- Command line
- File path
- File hash
- URL
- Domain
- Email sender
- Email recipient

Alert fields are critical for triage because they show who, what, where, and how.

---

## Example Alert: Double-Extension File Creation

Example alert name:

**Double-Extension File Creation**

This alert detects the creation of files that look like documents, videos, or images but are actually executable files.

Example:

**cats2025.mp4.exe**

This file may appear to the user as a video, but it is actually an executable.

Attackers often use double-extension files in phishing and malware delivery.

Example alert fields:

- Host: LPT-HR-009
- Process Name: chrome.exe
- Process User: S.Conway
- Target File: C:\Users\S.Conway\Downloads\cats2025.mp4.exe
- File URL: suspicious download source
- File MD5: file hash

SOC Analyst perspective:

This alert may indicate a phishing-related malware download.

The analyst should check whether the file was executed, whether EDR or antivirus generated additional alerts, whether the hash is known malicious, and whether the user or host shows more suspicious activity.

## Alert Prioritisation

Alert prioritisation is the process of deciding which alert should be handled first.

This is important when many alerts are waiting in the queue.

A SOC Analyst cannot investigate every alert at the same time.

The general prioritisation approach is:

1. Filter alerts
2. Sort by severity
3. Sort by time

---

## Step 1: Filter Alerts

First, filter the queue.

Do not take alerts that are already closed, already reviewed, or already assigned to another analyst.

Focus on alerts that are:

- New
- Unassigned
- Unresolved
- Not already investigated

This avoids duplicate work and prevents interference with other analysts.

---

## Step 2: Sort by Severity

After filtering, sort alerts by severity.

General priority order:

1. Critical
2. High
3. Medium
4. Low

Critical and high alerts usually have a greater chance of representing serious threats or high-impact activity.

Detection engineers usually assign severity based on likelihood and potential impact.

---

## Step 3: Sort by Time

If multiple alerts have the same severity, start with the oldest alert.

The reason is simple:

An older breach may have been active for longer.

If two alerts both represent real incidents, the attacker from the older alert may already be dumping data, moving laterally, or causing impact.

---

## Alert Prioritisation Rule

Practical L1 rule:

**Filter by status, sort by severity, then sort by time.**

This means:

- Ignore closed or assigned alerts
- Start with Critical
- Then High
- Then Medium
- Then Low
- If severity is the same, take the oldest alert first

---

## Alert Triage Workflow

Alert triage is also called:

- Alert handling
- Alert processing
- Alert investigation
- Alert analysis

The basic triage workflow:

1. Prioritise the alerts
2. Select the first alert
3. Assign the alert to yourself
4. Move the alert to In Progress
5. Read the alert name and description
6. Note key alert fields
7. Check whether a workbook or playbook exists
8. Investigate the activity in SIEM or EDR
9. Make a final verdict
10. Escalate if needed
11. Add an alert comment
12. Move the alert to Closed

---

## Initial Actions

Initial actions are designed to show ownership and avoid interfering with other analysts.

Before investigating deeply, a SOC L1 Analyst should:

- Assign the alert to themselves
- Move the alert to In Progress
- Read the alert name
- Read the alert description
- Review key alert fields

This means:

**This alert is now mine, and I am investigating it.**

---

## Investigation

Investigation is the most technical part of alert triage.

The analyst must decide whether the activity is benign, suspicious, or malicious.

Key investigation questions:

- Who is under threat?
- Which user is affected?
- Which host is affected?
- Which IP address is involved?
- What action triggered the alert?
- Was it a suspicious login, malware, phishing, or another activity?
- What happened before the alert?
- What happened after the alert?
- Are there related events?
- Is the behavior normal for this user or host?
- Is there another alert from EDR, SIEM, or firewall?
- Does threat intelligence show the IP, domain, URL, or hash as malicious?

---

## Workbooks, Playbooks, and Runbooks

Some SOC teams provide investigation instructions for specific alert types.

These instructions may be called:

- Workbook
- Playbook
- Runbook

They help analysts follow a consistent investigation process.

Example: Suspicious Login Workbook

Possible steps:

- Check source IP
- Check location
- Check device
- Check MFA activity
- Check impossible travel
- Check recent password reset
- Check mailbox forwarding rules
- Check whether the user confirms the activity
- Decide the verdict

If a workbook exists, the analyst should follow it.

If no workbook exists, the analyst should use general investigation logic.

---

## Final Actions

After investigation, the SOC Analyst must make a decision.

The analyst should decide whether the alert is:

- True Positive
- False Positive

If the alert is serious, unclear, or requires deeper analysis, the analyst should escalate it.

After the verdict, the analyst should add a clear comment explaining:

- What was reviewed
- What evidence was found
- Why the verdict was chosen
- Whether escalation is needed

Then the alert can be moved to Closed.

---

## Alert Commenting

A good alert comment should explain the analysis clearly.

Bad comment:

Checked. Closed.

Good comment:

Reviewed the alert details and related events. The activity involved a double-extension executable downloaded by user S.Conway on host LPT-HR-009. The file originated from a suspicious domain and may indicate malware delivery through phishing. No evidence of successful execution was confirmed in the available logs. Recommended further endpoint review and escalation if execution is detected.

A clear comment helps other analysts understand the decision.

It also supports auditability and future investigation.

---

## Analyst Summary

Alert triage is the core workflow of a SOC L1 Analyst.

An alert is generated when a security solution detects suspicious, anomalous, or rule-matching activity.

A SOC L1 Analyst should prioritise alerts, assign ownership, review alert details, investigate key fields, check related logs, make a verdict, document the reasoning, escalate when needed, and close the alert according to the SOC process.

---

## Interview Answer

When triaging an alert, I would first filter out closed, already reviewed, or assigned alerts. Then I would prioritise the remaining alerts by severity, starting with critical and high alerts. If multiple alerts have the same severity, I would start with the oldest one. After selecting an alert, I would assign it to myself, move it to In Progress, review the alert name, severity, description, and key fields such as user, host, IP address, file path, hash, or URL. If a workbook or playbook exists, I would follow it. Otherwise, I would investigate the activity in SIEM or EDR, review surrounding events, make a verdict such as True Positive or False Positive, add a clear analyst comment, escalate if needed, and close the alert according to the SOC process.

## Core Memory Lines

**Event → Log → Security Tool → Alert → Triage → Classification → Escalation or Closure**

**Raw log** = records what happened.

**Alert** = highlights activity that may require investigation.

**Alert fields** = evidence.

**Alert description** = rule explanation and triage guidance.

**Alert verdict** = investigation conclusion.

**Alert status** = workflow state.

**Assignee** = alert owner.

**True Positive** = real threat or suspicious malicious activity.

**False Positive** = no real threat.

**Prioritisation rule** = filter by status, sort by severity, then sort by time.

**Triage workflow** = Assign → In Progress → Read → Investigate → Verdict → Comment → Escalate if needed → Close.

**SOC L1 goal** = review alerts, distinguish benign from suspicious activity, document findings, and escalate real threats.
