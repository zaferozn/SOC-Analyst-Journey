# Case 003 - Wazuh SSH Brute-Force Investigation

## Executive Summary

This case documents the investigation of repeated SSH failed login activity observed in Wazuh. The objective was to determine whether the activity could represent a brute-force pattern and whether it required escalation.

The reviewed activity involved multiple SSH authentication failures on a monitored Ubuntu endpoint. The investigation focused on the event pattern, source IP address, targeted username, Wazuh alert fields, rule details, and whether any successful login followed the failed attempts.

This case was handled as a SOC investigation exercise. No confirmed compromise was identified based on the available evidence.

---

## Objective

The objective of this case was to investigate repeated SSH failed login activity in Wazuh and assess whether the observed pattern could indicate brute-force behavior.

The case focused on answering the following questions:

- What was observed?
- Where was it observed?
- Which source IP, username, host, or rule was involved?
- Why is the activity suspicious?
- What evidence supports the finding?
- What is the possible risk?
- What should be done next?
- Should the activity be escalated?

## Observed Activity

Repeated SSH failed login activity was observed in Wazuh. The events were related to authentication failures on the monitored Ubuntu endpoint.

The activity involved failed SSH login attempts against a non-existent user. Wazuh parsed the events as SSH authentication activity and classified the alert description as brute-force behavior against a non-existent user.

The activity was reviewed as a possible brute-force pattern because repeated SSH authentication failures may indicate credential guessing, unauthorized access attempts, or automated authentication activity.

At this stage, the activity was suspicious, but no confirmed compromise was identified.

## Evidence

Important analyst note:

## Analysis

The Wazuh alert showed repeated SSH failed login activity involving the invalid user `fakeuser` from source IP `192.168.64.1`. The activity was observed on the monitored Ubuntu endpoint `ubuntu-agent` and parsed by the `sshd` decoder.

The rule description classified the activity as brute-force behavior against a non-existent user. This is suspicious because repeated failed SSH attempts against invalid users may indicate credential guessing, automated scanning, or unauthorized access attempts.

However, this case was reviewed in a controlled lab environment. The source IP address `192.168.64.1` appears to be part of the local lab network. Therefore, the activity should be documented as a possible brute-force pattern, not as a confirmed compromise.

No successful login was confirmed from this evidence alone. A successful SSH login after repeated failed attempts would increase the severity and may require escalation.

---

## Risk

The possible risk is unauthorized SSH access if the attacker successfully guesses valid credentials or targets an existing user account.

If repeated failed SSH attempts are followed by a successful login, the risk may include:

- credential compromise
- unauthorized remote access
- account takeover
- lateral movement from the compromised host
- further reconnaissance or privilege escalation

In this case, the risk was reviewed as suspicious authentication activity, but no confirmed compromise was identified based on the available evidence.

---

## Recommended Next Steps

Recommended next steps:

1. Confirm whether the source IP `192.168.64.1` is expected in the lab environment.
2. Check whether the targeted username `fakeuser` exists on the monitored host.
3. Review the number of failed SSH attempts within the selected time window.
4. Check whether any successful SSH login occurred after the failed attempts.
5. Review whether the same source IP targeted other usernames.
6. Review whether the activity repeated across different time windows.
7. Escalate if the source IP is unauthorized, multiple usernames are targeted, privileged accounts are targeted, or a successful login follows the failed attempts.

---

## Escalation Decision

Escalation status: Not escalated in this lab environment.

Reason:

The activity was generated and reviewed in a controlled lab environment. Although Wazuh classified the activity as brute-force behavior, no confirmed successful login or compromise was identified from the available evidence.

The activity should be escalated in a real SOC environment if the source IP is unauthorized, if multiple usernames are targeted, if privileged accounts are involved, or if a successful SSH login occurs after repeated failed attempts.

## Lessons Learned

This case improved my understanding of how repeated SSH failed login activity appears in Wazuh and how an L1 SOC analyst should review alert details before making an escalation decision.

Key lessons from this case:

- Wazuh alert details provide important investigation fields such as `agent.name`, `agent.ip`, `data.srcip`, `data.srcuser`, `decoder.name`, `location`, `rule.description`, and `full_log`.
- A SIEM rule description may classify activity as brute-force, but the analyst still needs to validate the context.
- Repeated failed SSH login attempts against invalid users may indicate credential guessing or automated scanning.
- A successful login after repeated failed attempts would increase the severity of the case.
- Controlled lab activity should be clearly documented and not reported as a confirmed compromise.
