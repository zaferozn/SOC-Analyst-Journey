# Lab 002 - Fresh SSH Authentication Baseline

## Objective

This lab documents a fresh SSH authentication baseline review in a controlled home SOC lab environment.

The goal was to generate new SSH authentication events and review failed logins, successful logins, invalid username activity, source IP information, and a short authentication timeline.

## Environment / Data Source

Host: `soclab`  
Target system: Ubuntu Linux VM  
VM IP address: `192.168.64.7`  
Source IP observed: `192.168.64.1`  
Tool: `journalctl`, `grep`, `wc`, `sort`, `uniq`, `tail`  
Log source: Linux journal logs  
Service: SSH  

## Event Generation

A controlled failed-login test was generated by attempting to authenticate with the invalid username `fakeuser`.

A successful SSH login was then generated by authenticating with the valid user `analyst`.

This created a simple authentication baseline containing both failed and successful SSH activity.

## Commands Used

```bash
journalctl --since "15 minutes ago" | grep -i "Failed password" | wc -l
journalctl --since "15 minutes ago" | grep -i "Accepted password" | wc -l
journalctl --since "15 minutes ago" | grep -i "Invalid user" | wc -l
journalctl --since "15 minutes ago" | grep -Ei "Failed password|Accepted password|Invalid user" | grep -oE 'from [0-9]+\.[0-9]+\.[0-9]+\.[0-9]+' | sort | uniq -c
journalctl --since "15 minutes ago" | grep -Ei "Failed password|Accepted password|Invalid user" | tail -n 30
```

## Observed Results

- Failed password entries: `3`
- Accepted password entries: `1`
- Log lines containing `Invalid user`: `5`
- Source IP observed: `192.168.64.1`
- Invalid username observed: `fakeuser`
- Valid username observed: `analyst`

## Field Interpretation

| Field | Value | Meaning |
|---|---|---|
| Host | `soclab` | Linux VM where SSH logs were collected |
| Source IP | `192.168.64.1` | System initiating the SSH connection |
| Target IP | `192.168.64.7` | Linux VM receiving the SSH connection |
| Invalid username | `fakeuser` | Username that does not exist on the Linux system |
| Valid username | `analyst` | Existing local user account |
| Event type | `Failed password` | Failed SSH authentication attempt |
| Event type | `Accepted password` | Successful SSH authentication |
| Source port | `57213`, `57215` | Temporary client-side source ports |
| Protocol | `ssh2` | SSH protocol version 2 |

## Analysis

The logs showed `3` failed SSH login attempts targeting the invalid username `fakeuser`.

The activity came from the source IP `192.168.64.1`, which was the host machine in the controlled lab environment.

A successful SSH login for the valid user `analyst` occurred shortly after the failed attempts.

In this controlled lab, the successful login was expected. In a real SOC environment, a successful login after repeated failed attempts should be reviewed to determine whether it was expected and authorized.

## Important Observation

The `Invalid user` count does not necessarily represent separate failed login attempts.

In this lab, the command counted log lines containing the phrase `Invalid user`. This included different types of SSH log messages, such as:

- Initial invalid user detection
- Failed password entries for an invalid user
- Connection closed messages involving an invalid user

For this reason, `Failed password` entries and `Invalid user` log lines should be interpreted separately.

## Pattern Identified

The observed authentication pattern was:

```text
Invalid user detected → repeated failed password attempts → connection closed → successful login by valid user
```

In this lab, this pattern was expected because the events were manually generated.

In a real SOC environment, this pattern may require further review because a successful login after failed attempts can indicate possible credential guessing followed by valid authentication.

## Risk

The main risk in a real environment would be unauthorized access through valid credentials after repeated failed authentication attempts.

If the successful login was not expected, the activity could indicate that an attacker guessed or obtained valid credentials.

## Recommended Next Steps

- Confirm whether the successful login was expected.
- Check whether the source IP is known and authorized.
- Review whether the valid user account should normally connect through SSH.
- Compare the failed and successful login timestamps.
- Review session activity after the successful login.
- Escalate if the successful login cannot be confirmed as legitimate.

## Final SOC Summary

A fresh SSH authentication baseline was reviewed in a controlled Linux lab environment. The observed activity included three failed SSH login attempts against the invalid username `fakeuser`, followed by one successful SSH login for the valid user `analyst`. All activity originated from the source IP `192.168.64.1`. In this lab, the activity was expected. In a real SOC environment, a successful login following repeated failed attempts should be reviewed to determine whether the access was authorized.

## Lessons Learned

This lab showed how to build a basic SSH authentication baseline by comparing failed logins, successful logins, invalid user activity, source IP information, and authentication timelines. It also demonstrated that log-line counts must be interpreted carefully because some phrases may appear in multiple SSH message types.
