# Lab 008 - Wazuh Authentication Event Review and First Alert Triage

## Executive Summary

This lab documents the review of authentication-related Wazuh events generated by an enrolled Ubuntu agent. The objective was to identify Linux authentication events in the Wazuh Threat Hunting module and understand how they support first-level SOC triage.

The observed events included SSH authentication success, missed password activity, non-existent user login attempts, PAM login failure, and PAM session activity. The lab focused on reading Wazuh rule descriptions, rule IDs, severity levels, timestamps, and agent information before making any escalation decision.

This lab was handled as a first-level event review exercise, not as a confirmed security incident.

## Objective

The objective of this lab was to review Wazuh authentication events from a monitored Ubuntu endpoint and understand how failed login activity, successful authentication, and session events appear in a SIEM.

## Environment / Data Source

Host: ubuntu-agent  
Wazuh server: wazuh-server

Tool: Wazuh Threat Hunting  
Log source: Linux authentication logs / Wazuh alerts

## Observed Activity

Authentication-related events were observed from the monitored Ubuntu endpoint in Wazuh Threat Hunting.

The observed activity included:

- SSH authentication success
- Failed login activity
- A user missing the password more than one time
- Login attempts using a non-existent user
- PAM login failure
- PAM session opening

These events show how Wazuh converts Linux authentication activity into structured security events.

## Analysis

The observed Wazuh events represent different stages of Linux authentication activity.

The non-existent user login attempt may indicate a mistyped username, automated scanning, or credential guessing. The missed password event may indicate user error, but it may also suggest repeated failed authentication attempts. The PAM login failure confirms that at least one login attempt failed.

The SSH authentication success and PAM session opened events indicate that a login session was successfully created. In SOC triage, this becomes more important when successful authentication occurs after failed login attempts from the same source IP address.

At this stage, the activity should not be classified as a confirmed incident because the source IP address, targeted username, and full log context were not fully reviewed in this lab. However, the event sequence is important for learning how to identify a possible failed-to-successful authentication pattern.

## Risk

If failed login attempts, non-existent user attempts, and successful authentication events are connected to the same source IP address, the activity may indicate credential guessing or unauthorized access.

If successful authentication is followed by privileged activity, the risk level increases because the user may have gained access and then performed administrative actions.

## Recommended Next Steps

- Open individual Wazuh events and review the full_log field.
- Identify the source IP address involved in the failed login attempts.
- Identify the targeted username.
- Check whether the same source IP appears across failed and successful authentication events.
- Compare the Wazuh alert with the original Linux authentication log.
- Review whether sudo or privileged activity occurred after the successful login.
- Escalate only if repeated attempts, suspicious source IP activity, or unauthorized successful login is confirmed.

## Lessons Learned

This lab improved my understanding of how Wazuh represents Linux authentication activity as structured SIEM events. I learned that a SOC analyst should not rely only on the alert title. The analyst must review the agent name, rule description, rule ID, rule level, timestamp, full log, source IP address, and user context before making an escalation decision.

## Raw Log vs Wazuh Event Understanding

In the previous lab, I reviewed SSH authentication activity through raw log data in Discover. In this lab, I reviewed the same type of activity through Wazuh Threat Hunting events.

The main difference is that raw logs show the original event context, while Threat Hunting events show parsed and structured SIEM data such as rule.id, rule.level, rule.description, agent.name, and timestamp.

This helped me understand how raw authentication logs are converted into Wazuh security events for first-level SOC triage.
